# ClawOps Contractor Performance Review Framework

## Purpose

This framework ensures contractors know what success looks like, receive regular feedback, and have clear paths for growth. We review performance to:

1. Recognize great work and reward it
2. Catch problems early and fix them
3. Help contractors grow their skills
4. Make data-driven decisions about long-term partnerships

---

## Review Schedule

### Week 1: Onboarding Check-In
- **Timing:** End of first week
- **Format:** 15-min call + written feedback
- **Focus:** Are they set up for success? Early red flags?

### 30-Day Review
- **Timing:** After 4 weeks of work
- **Format:** Structured review (see template below)
- **Focus:** Quality, speed, communication, culture fit
- **Outcome:** Continue, adjust scope, or part ways

### 90-Day Review
- **Timing:** After 3 months (if still engaged)
- **Format:** Comprehensive review + goal-setting
- **Focus:** Long-term potential, rate adjustment, expanded role
- **Outcome:** Renew, promote, or transition out

### Ongoing: Weekly Check-Ins
- **Timing:** Every Monday (or first work day of week)
- **Format:** Quick async update or 15-min call
- **Focus:** Current projects, blockers, quick wins

---

## Performance Dimensions

We evaluate contractors across 5 dimensions. Each scored 1-5:

1. **Quality** - Does the work meet standards?
2. **Speed** - Do they deliver on time?
3. **Communication** - Are they clear, responsive, proactive?
4. **Autonomy** - Do they need hand-holding or figure things out?
5. **Culture Fit** - Do they align with our values and work style?

### Scoring Rubric

**5 = Exceptional** - Exceeds expectations, sets new standard
**4 = Strong** - Consistently meets expectations, occasional excellence
**3 = Solid** - Meets expectations, reliable
**2 = Needs Improvement** - Gaps in performance, requires support
**1 = Unacceptable** - Not meeting basic requirements

## Key Performance Indicators (KPIs)

### For Python/Automation Developers

**Quality Metrics:**
- Code passes internal QA first time: >80%
- Client revision requests: <2 per project
- Bugs reported post-delivery: <1 per month
- Code review feedback: Positive trend over time

**Speed Metrics:**
- Projects delivered on time: >90%
- Estimated vs actual hours: Within 20%
- Response time to urgent bugs: <2 hours

**Communication Metrics:**
- Daily updates posted: 100% of work days
- Response time to Slack: <4 hours during work hours
- Proactive problem flagging: Evidence in updates

**Autonomy Metrics:**
- Questions asked per project: Decreasing over time
- Self-correction rate: Catches own mistakes before submission

### For Virtual Assistants/Project Coordinators

**Quality Metrics:**
- Client satisfaction (informal check-ins): >4.5/5
- Meeting notes completeness: 100% (all action items captured)
- File organization: No missing documents or duplicates
- Invoice accuracy: 100% (no payment errors)

**Speed Metrics:**
- Client response time: <4 hours during business hours
- Task completion rate: >95% on time
- Follow-up rate: 100% (no dropped balls)

**Communication Metrics:**
- Daily updates posted: 100% of work days
- Email clarity: <10% require clarification
- Proactive status updates: Weekly to Blake

**Autonomy Metrics:**
- Problems solved without escalation: Increasing over time
- Process improvements suggested: At least 1 per month

### For Technical Writers

**Quality Metrics:**
- Client acceptance first draft: >70%
- Documentation tested by non-author: 100% pass rate
- Readability score (Hemingway App): Grade 8 or lower for end-user docs
- Accuracy: Zero factual errors post-publication

**Speed Metrics:**
- Drafts delivered on time: >90%
- Turnaround on revisions: <24 hours

**Communication Metrics:**
- Response time: <4 hours during work hours
- Clarifying questions asked upfront: Before starting (not mid-draft)

**Autonomy Metrics:**
- Independent research: Can write docs without step-by-step instructions
- Initiative: Suggests improvements to existing docs without prompting

## Feedback Cycles

### Real-Time Feedback (Ongoing)

**When:** Immediately after a significant win or mistake  
**Format:** Quick Slack message or 5-min call  
**Purpose:** Reinforce good behavior, correct course quickly

**Examples:**
- "Great catch on that bug in the API code. That saved us a client escalation."
- "Your draft was late by 2 days without a heads-up. Let's talk about how to communicate delays proactively."

### Project-Level Feedback (Per Project)

**When:** After project delivery  
**Format:** Written feedback in project tracker + optional 10-min call  
**Purpose:** Specific performance data for review cycles

**Template:**
```
Project: [Name]
Contractor: [Name]
Quality: [1-5] - [Notes]
Speed: [1-5] - [Notes]
Communication: [1-5] - [Notes]
Overall: [Notes]
```

### Formal Reviews (30/90 Days)

**When:** Scheduled milestones  
**Format:** Structured document + 30-min call  
**Purpose:** Comprehensive assessment, goal-setting, decisions

## Performance Improvement Plans (PIP)

### When to Use a PIP

**Trigger:** Score of 2 or lower in any dimension for 2+ consecutive weeks  
**Purpose:** Give contractor clear path to improve before termination  
**Duration:** 2 weeks

### PIP Structure

**Document Includes:**
1. Specific performance gaps (with examples)
2. Required improvements (measurable)
3. Support offered (resources, check-ins, adjusted scope)
4. Timeline (usually 2 weeks)
5. Consequences if improvements not met (termination)

**Example PIP:**

**Issue:** Quality scores have been 2/5 for past 3 weeks (bugs in delivered code)

**Required Improvements:**
- Deliver next 2 projects with zero bugs in QA
- Add test coverage to all critical functions
- Submit self-QA checklist with each project

**Support Offered:**
- Pair programming session with senior dev
- Access to testing tools/tutorials
- Daily 10-min check-ins for questions

**Timeline:** 2 weeks (next 2 projects)

**Outcome if Successful:** Continue partnership, rate remains same  
**Outcome if Unsuccessful:** Part ways, final payment processed

## Rewards & Recognition

### Performance-Based Bonuses

**Criteria:**
- 5/5 rating in quality for 3+ consecutive projects: $500 bonus
- Client specifically praises contractor: $250 bonus
- Contractor refers another excellent contractor who passes trial: 10% of first month revenue from that contractor

### Rate Increases

**Trigger:** Consistently strong performance (4+ overall) for 90 days  
**Amount:** $5-10/hr increase, negotiated based on market rate + value delivered

### Expanded Opportunities

**High Performers Get:**
- First access to larger, more interesting projects
- Opportunity to mentor other contractors
- Consideration for full-time role if we hire in-house
- Invitations to strategic planning discussions

### Public Recognition

- Shout-outs in team Slack
- Featured in case studies (with permission)
- LinkedIn recommendations

## Termination Process

### When to Part Ways

**Immediate Termination (Rare):**
- Breach of contract (sharing client data, violating NDA)
- Fraud (billing for unworked hours)
- Gross misconduct (abusive communication)

**Performance-Based Termination:**
- Fails to meet PIP requirements
- Overall score <3 for 2+ consecutive reviews
- Consistent reliability issues (missing deadlines, unresponsive)

### Off-Boarding Procedure

1. **Notification:** 1 week notice (unless immediate termination required)
2. **Transition:** Complete in-progress work or hand off to another contractor
3. **Access Removal:** Revoke tool access on last day
4. **Final Payment:** Process within 3 business days
5. **Exit Interview:** Optional 15-min call to gather feedback

**Tone:** Professional, respectful, no burning bridges  
**Example Script:** "We have decided to move in a different direction. Your last day will be [date]. We will process your final payment by [date]. Thank you for your contributions, and we wish you well."

## Continuous Improvement

### Quarterly Framework Review

**Timing:** Every 3 months  
**Owner:** Avery (CHRO) + Blake (CEO)  
**Purpose:** Refine KPIs, update rubrics based on what we have learned

**Questions to Ask:**
- Are our KPIs predictive of success?
- Are reviews taking too much time?
- What patterns do we see in high vs low performers?
- What should we add/remove from this framework?

---

## Success Metrics for This Framework

**We will know this framework is working if:**

1. **Retention:** >80% of contractors pass 30-day review and continue
2. **Quality:** Average contractor rating improves from 30-day to 90-day review
3. **Efficiency:** Reviews take <1 hour per contractor to complete
4. **Predictability:** PIP success rate >50% (we catch issues early enough to fix)
5. **Satisfaction:** Contractors report framework is fair and helpful (exit interview data)

## Appendix: Sample Feedback Phrases

### Positive Feedback Examples

**Quality:**
- "Your code is consistently clean and well-documented. I can hand it to anyone on the team."
- "Zero revisions on the last 3 projects. That is exceptional."

**Speed:**
- "You have delivered every project on time for 6 weeks straight. Reliable."
- "You under-promised and over-delivered. Love the proactive communication."

**Communication:**
- "Your daily updates are the best on the team. Clear, concise, no fluff."
- "You flagged that issue 2 days before it became a problem. Great anticipation."

**Autonomy:**
- "You solved that integration issue without needing my help. That is what I like to see."
- "You have gone from 10 questions per project to 2. Clear growth."

### Constructive Feedback Examples

**Quality:**
- "The last 2 projects had bugs that QA caught. Let's add a self-test checklist before submission."
- "Your code works, but it is hard to read. Let's focus on variable naming and comments."

**Speed:**
- "You have been late on 3 of the last 5 deadlines. What is getting in the way?"
- "Your estimates are consistently 2x actual time. Let's work on more accurate scoping."

**Communication:**
- "I am seeing 8-hour response gaps on Slack. Clients need faster replies during your work hours."
- "Your updates say 'working on it' but do not tell me progress. More detail helps."

**Autonomy:**
- "You are asking the same types of questions every project. Let's document answers so you can self-serve."
- "I need you to try solving it first before asking. Show me what you have tried."
