# Objection Handling

Common objections and how to respond to them effectively.

---

## Pricing Objections

### "It's too expensive"

**What they really mean:** They don't see enough value to justify the price.

**Response framework:**
1. Acknowledge the concern
2. Reframe around ROI
3. Provide comparison

**Example response:**
```
I understand - it's a real investment. Let me put it in perspective:

If this automation saves your team [X hours per week], that's
[X x hourly rate] per month in labor costs alone. The project
pays for itself in [Y weeks/months].

We could also look at a smaller scope to start - our Starter
tier at $[X] covers [specific subset]. Once you see the results,
we can expand from there.
```

**If they're genuinely budget-constrained:**
- Offer a smaller scope (Starter tier)
- Suggest milestone-based payments
- Propose a pilot project that demonstrates value
- Never go below 50% margin - walk away instead

### "We can build it ourselves"

**What they really mean:** They underestimate the complexity or overestimate their capacity.

**Response:**
```
Totally valid option! A few things to consider:

- How many hours would your team spend? At $[internal rate]/hr,
  that's $[X] in opportunity cost
- Your team's time away from [their core product/business]
- We've built this type of system [X] times - we know the
  pitfalls and shortcuts

If you do build internally, happy to advise. If you'd rather
ship in [timeline] and keep your team focused on [core business],
we're ready to go.
```

### "Competitor X is cheaper"

**Response:**
```
That's worth looking into. A few questions to compare fairly:

- Do they include [specific feature we include]?
- What's their support period? (Ours is [30/60] days)
- How quickly can they deliver? (We deliver in [timeline])
- Do they have experience with [specific technology/approach]?

Price is one factor, but total value matters more. Happy to
do a side-by-side comparison if that would help.
```

## Timing Objections

### "Not the right time"

**What they really mean:** It's not a priority, or they're overwhelmed.

**Response:**
```
Totally understand. When would be a better time to revisit this?

In the meantime, I'd suggest [one small action they can take now
to prepare]. That way, when you're ready, we can hit the ground
running.

I'll follow up in [suggested timeframe] if that works?
```

### "We need to think about it"

**Response:**
```
Of course - it's a real decision. To help your thinking:

- What specific concerns should I address?
- Is there information you need that I haven't provided?
- Would it help to talk through [specific aspect] in more detail?

Happy to answer any questions that come up. I'm here whenever
you're ready.
```

### "Our priorities changed"

**Response:**
```
That happens. What shifted?

[If their new priority is something we can help with, pivot]

If [original project] comes back on the radar, the approach
we discussed would still work. Let me know when the time is right.
```

## Trust Objections

### "We haven't worked with you before"

**Response:**
```
Fair concern. A few ways to reduce the risk:

- Start with a Starter project ($500-$1,500) so you can evaluate
  our work with minimal commitment
- I can share [relevant examples/case studies/portfolio work]
- We include [30/60] days of support, so you're covered after delivery
- Payment is milestone-based, so you only pay for completed work

Would a small pilot project make sense as a first step?
```

### "How do we know this will work?"

**Response:**
```
Great question. Here's how we ensure success:

1. We start with a clear scope and success criteria (agreed by both of us)
2. I keep you updated throughout - no surprises
3. [X] rounds of revisions are included
4. [30/60]-day support period for any issues
5. If something isn't working, we adjust during the project

We can also define specific success metrics upfront so there's
no ambiguity about what "working" means.
```

### "Can you provide references?"

**Response:**
```
I appreciate you doing due diligence. [Choose one]:

Option A (if references available):
"Absolutely. I'll connect you with [client name] who had a
similar project. They can speak to our work firsthand."

Option B (if early stage):
"We're a newer team, so formal references are still building.
What I can offer is: a small pilot project at reduced risk,
detailed examples of our technical capabilities, and a money-back
guarantee if the pilot doesn't meet agreed criteria."
```

## Scope Objections

### "Can you add [X] to the project?"

**Response:**
```
Great idea! [X] would definitely add value. There are two ways
to handle it:

1. Include it now: The project scope and price would adjust to
   $[new amount] with a [new timeline]
2. Phase 2: We deliver the current scope first, then add [X]
   as a follow-up project

Which approach works better for you?
```

**Never:** Add scope without adjusting price. Scope creep kills margins.

### "We need it faster"

**Response:**
```
I can explore expediting. To deliver by [their date], I'd need to:

Option A: Reduce scope to [core deliverables only] - same price
Option B: Full scope, rush delivery - [X]% premium ($[amount])
Option C: Deliver MVP by [their date], full version by [our date]

Which option works best for your situation?
```

## Competition Objections

### "We're evaluating other options"

**Response:**
```
Smart approach. To help your evaluation:

- What criteria are most important to you? (Speed, price, quality, support)
- I'm happy to answer any specific comparison questions
- [Optional: share a differentiator they might not have considered]

Whatever you decide, I want to make sure you have all the info
you need to compare fairly.
```

## The Nuclear Objections

### "We decided to go with someone else"

**Response:**
```
Appreciate you letting me know. Best of luck with the project!

If I could ask: what was the deciding factor? Understanding helps
me improve. No hard feelings either way.

If things don't work out or you need additional help down the road,
my door is always open.
```

Log the reason. Learn from it. Move on.

### "We're not interested"

**Response:**
```
Understood. Thanks for your time.

If AI automation ever becomes a priority for [Company], feel free
to reach out. Wishing you all the best!
```

Do not follow up further. Respect the no.

---

## Objection Handling Rules

1. **Listen fully** before responding
2. **Acknowledge** their concern genuinely
3. **Never argue** or get defensive
4. **Ask clarifying questions** to understand the real objection
5. **Reframe** around value, not features
6. **Offer options** (give them control)
7. **Know when to walk away** (some deals aren't meant to close)
8. **Document** objections for pattern analysis
